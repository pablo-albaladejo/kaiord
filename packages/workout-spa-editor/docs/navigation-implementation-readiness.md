## Needs a design step

**WP-11 — back-nav origin source (the canonical missing mechanism).** Confirmed by code: `grep "from=" src/ = 0 matches`; no `history.back()`/`useHistoryState` in app code; `src/routing/` holds only `picker-href.ts` + `constants.ts`. Every #4/#19/#28 fix depends on a source that does not exist.

- _Smallest unblock:_ an explicit **`?from=<origin>` query contract** + a new `src/routing/resolve-back-target.ts` resolver (testable under the existing `memoryLocation` harness; deterministic; no deep-link app-exit). Reject `history.back()` and wouter `useHistoryState()` — the adversary confirmed both are untestable under `memoryLocation({record:true})`, which tracks only the path list, not `window.history.state`. Define origin vocabulary (`library | calendar | calendar-day | coaching | today | detail`) before coding. Then thread `?from=` through ~12+ entry call sites and rewrite 3 pinned test files (`picker-href.test.ts`, `EditorPage.test.tsx`, `WorkoutDetail.test.tsx`). Keep the no-origin edit-mode case rendering no back button to preserve the existing pin.

**WP-5 — scratch persist-on-date (#3 / D1).** Confirmed: scratch's only save is a file export (`save-handler.ts` → `exportWorkout` + `downloadWorkout`, never `workouts.put`); `ScratchEditorSurface.test.tsx:159` pins "no Dexie write on mount."

- _Smallest unblock:_ author a new `persist-scratch-workout.ts` mirroring `persist-imported-workout.ts` (date + profileId + sport → `WorkoutRecord` → `workouts.put` → navigate `/workout/:id`), fired on an explicit save action (mount stays side-effect-free). Decide its home (a scratch-local control, since `WorkoutActions` is shared with the id-loaded editor). **Drop the literal "Save as draft (unscheduled)"** — the `date` field is `z.iso.date()` non-nullable AND a primary Dexie index (`[profileId+date]`, `[date+state]`, plain `date`, confirmed `dexie-schemas.test.ts:28`), so an unscheduled record needs a schema + index + migration and still wouldn't appear in the date-indexed calendar. (The probe's EditorPage 80-line cap fear is **unfounded** — `pages/**/*.tsx` is capped at **150** per `eslint.config.js:253-257`; EditorPage is 94 lines.) **#1, #5, #37** within WP-5 are ready (#37 just needs a small `isValidCalendarDate` round-trip helper — none exists today).

**WP-9 — settings section anchors + focus (D4).** Confirmed missing: `SettingsPage.tsx:18` reads only `useParams(tab)`; `useFocusOnRouteChange` keys on pathname-only and targets the single route `<h1>`. And **"Export everything" has no target block** (`PrivacyTab` has only Analytics + Data Management; audit:392 "no export surface exists").

- _Smallest unblock:_ build a `useFocusOnSectionChange` hook that reads `?section=` via `useSearch` and focuses the target via the existing `applyFocusToElement` helper; add `id`+`tabIndex={-1}` to AI sub-sections.
