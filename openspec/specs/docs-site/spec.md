# Docs Site

## Requirements

### Requirement: Documentation site serves at /docs/ path

The documentation site SHALL be served at `kaiord.com/docs/` as a VitePress static site built from `packages/docs/`.

#### Scenario: User visits docs

- **WHEN** a user navigates to `https://kaiord.com/docs/`
- **THEN** the VitePress documentation site loads with the docs home page

#### Scenario: Docs are static

- **WHEN** the docs site is built
- **THEN** the output SHALL be static HTML with no server-side rendering

### Requirement: Docs home page with onboarding flow

The docs home page (`index.md`) SHALL display a hero with a one-liner value proposition, a prominent "Quick Start" CTA, a 3-step visual (Install → Convert → Done), and feature cards linking to Formats, CLI, MCP, and API Reference.

#### Scenario: New developer onboarding

- **WHEN** a developer lands on `/docs/`
- **THEN** they SHALL see a clear path to the Quick Start guide within the first viewport

### Requirement: Quick Start tutorial

The docs SHALL include a `guide/quick-start.md` page that takes a developer from `pnpm add @kaiord/core @kaiord/fit @kaiord/tcx` to a working FIT-to-TCX conversion in 10 steps or fewer (prerequisites: Node 18+, pnpm installed). This SHALL be the first item in the sidebar.

#### Scenario: Quick Start outcome preview

- **WHEN** the Quick Start page loads
- **THEN** the first paragraph SHALL describe the concrete end result ("By the end of this guide, you'll convert a Garmin FIT file to TCX in 4 lines of TypeScript")

#### Scenario: Quick Start is self-contained

- **WHEN** a developer follows the Quick Start page
- **THEN** they SHALL be able to run a working conversion without reading any other page

### Requirement: Why Kaiord page

The docs SHALL include a `guide/why-kaiord.md` page that frames the problem space (fragmented fitness formats, no unified TypeScript SDK), includes search-friendly terms ("convert FIT file TypeScript", "Garmin workout API"), and positions Kaiord against alternatives with a comparison table or positioning statement.

#### Scenario: Search discoverability

- **WHEN** the Why Kaiord page is indexed
- **THEN** it SHALL contain high-value search terms that drive organic traffic

#### Scenario: Competitive positioning

- **WHEN** a developer reads the Why Kaiord page
- **THEN** they SHALL understand how Kaiord differs from parsing libraries individually (fitparse, gpx-parser, etc.)

### Requirement: Manual guides

The docs site SHALL include manually written guides migrated from existing documentation: Getting Started, Architecture, KRD Format, Testing, and Contributing.

#### Scenario: Guide content accessible

- **WHEN** a user navigates to `/docs/guide/getting-started`
- **THEN** the Getting Started guide SHALL be displayed with installation instructions and basic usage examples

### Requirement: Auto-generated API reference

The docs site SHALL include API reference pages auto-generated by TypeDoc from TSDoc comments across all public packages. Generated files SHALL be gitignored and regenerated at build time.

#### Scenario: API reference reflects current code

- **WHEN** a developer adds a new exported function with TSDoc to `@kaiord/core`
- **THEN** the next docs build SHALL include that function in the API reference without manual intervention

### Requirement: Format adapter documentation

The docs site SHALL include pages for each format adapter (KRD, FIT, TCX, ZWO, GCN) documenting usage, capabilities, and limitations.

#### Scenario: Format docs accessible

- **WHEN** a user navigates to `/docs/formats/fit`
- **THEN** documentation for the FIT format adapter SHALL be displayed including import examples and supported fields

### Requirement: CLI command reference

The docs site SHALL include a CLI command reference page documenting all available commands, flags, and usage examples.

#### Scenario: CLI reference accessible

- **WHEN** a user navigates to `/docs/cli/commands`
- **THEN** all CLI commands (convert, validate, inspect, diff, extract-workout, garmin) SHALL be documented

### Requirement: MCP tool reference

The docs site SHALL include an MCP tool reference page documenting all available tools, resources, and prompts.

#### Scenario: MCP reference accessible

- **WHEN** a user navigates to `/docs/mcp/tools`
- **THEN** all MCP tools (convert, validate, inspect, diff, extract-workout, list-formats) SHALL be documented

### Requirement: Dark mode and branding

The docs site SHALL use dark mode by default with VitePress theme toggle (persisted via localStorage), display the Kaiord logo in the sidebar/nav, use the shared brand color tokens, system font stack (no web fonts), and include "Built by Pablo Albaladejo" attribution.

#### Scenario: Consistent branding

- **WHEN** the docs site loads
- **THEN** the site SHALL use dark mode, display the Kaiord logo, and match the landing page's visual style

### Requirement: Search with keyboard shortcut

The docs site SHALL include full-text local search powered by VitePress MiniSearch, accessible via Cmd+K / Ctrl+K. Search SHALL index all pages including auto-generated API reference. Zero-results state SHALL show fallback links to Quick Start and Formats overview.

#### Scenario: Keyboard search

- **WHEN** a user presses Cmd+K (or Ctrl+K)
- **THEN** the search modal SHALL open and accept queries

#### Scenario: Zero results

- **WHEN** a search query returns no results
- **THEN** fallback links to Quick Start and Formats overview SHALL be displayed

### Requirement: LLM-friendly output

The docs site SHALL generate an `llms.txt` file at build time using `vitepress-plugin-llms`. The llms.txt SHALL contain sufficient context for an LLM to answer API questions about Kaiord accurately.

#### Scenario: llms.txt accessible

- **WHEN** an AI tool fetches `https://kaiord.com/docs/llms.txt`
- **THEN** a structured text file with documentation content SHALL be returned

#### Scenario: LLM can answer questions from llms.txt

- **WHEN** an LLM is given only the llms.txt content and asked "How do I convert a FIT file to TCX using Kaiord?"
- **THEN** the answer SHALL be correct and not hallucinated

### Requirement: Per-page SEO

Every documentation page SHALL include a unique `<title>`, `<meta name="description">`, `og:title`, `og:description`, `og:image`, and `<link rel="canonical">` via VitePress frontmatter and `transformPageData` hook. The `<html>` element SHALL include `lang="en"`. Auto-generated TypeDoc pages SHALL have descriptions extracted from TSDoc first sentences.

#### Scenario: Meta tags present

- **WHEN** any doc page is rendered
- **THEN** it SHALL have a unique title, meta description, and per-page OG title/description — not generic or empty

#### Scenario: Lang attribute

- **WHEN** any doc page is rendered
- **THEN** the `<html>` element SHALL include `lang="en"`

### Requirement: Auto-generated sitemap

VitePress SHALL auto-generate `sitemap.xml` for all doc pages using the built-in `sitemap` config. This SHALL be merged with the landing sitemap at deploy time.

#### Scenario: All doc pages in sitemap

- **WHEN** the sitemap is generated
- **THEN** every doc page (guides, formats, API, CLI, MCP) SHALL have an entry

### Requirement: Mobile navigation

The docs sidebar SHALL collapse to a hamburger menu below 768px. Sidebar sections SHALL be collapsible. Search SHALL be accessible from mobile nav. Touch targets SHALL meet 44x44px minimum.

#### Scenario: Mobile sidebar

- **WHEN** the viewport is below 768px
- **THEN** the sidebar SHALL collapse to a hamburger and sections SHALL be collapsible

### Requirement: Performance budget

The docs site SHALL achieve Lighthouse Performance >= 90 on mobile. LCP SHALL be under 1.5s. CLS SHALL be under 0.05. Total JS SHALL be under 150KB gzipped. The site SHALL use system fonts only (no web font downloads). MiniSearch index SHALL be lazy-loaded.

#### Scenario: Lighthouse passes

- **WHEN** Lighthouse is run on the deployed docs
- **THEN** Performance SHALL be >= 90 and no web font downloads SHALL be detected

### Requirement: Code block copy button

All code blocks in documentation SHALL have a copy-to-clipboard button. VitePress built-in copy button SHALL be enabled.

#### Scenario: Copy code example

- **WHEN** a user hovers over a code block
- **THEN** a copy button SHALL appear and clicking it copies the code to clipboard

### Requirement: On-this-page outline

Documentation pages with 3 or more headings SHALL display a sticky "On this page" outline sidebar via VitePress `outline` config.

#### Scenario: Outline visible on long pages

- **WHEN** a user views a page with 3+ headings (e.g., Architecture guide)
- **THEN** a sticky outline sidebar SHALL be visible on desktop showing heading links

### Requirement: Docs 404 page

VitePress SHALL serve a custom 404 page for non-existent `/docs/` routes with a search box and links to Quick Start and docs home.

#### Scenario: Docs 404

- **WHEN** a user navigates to `/docs/nonexistent`
- **THEN** a helpful 404 page SHALL display with navigation back to docs home and search

### Requirement: JSON-LD structured data

Each documentation page SHALL include JSON-LD structured data: `BreadcrumbList` for navigation hierarchy and `TechArticle` with author (Pablo Albaladejo). The docs home SHALL include `WebSite` schema with `SearchAction`.

#### Scenario: Structured data present

- **WHEN** a doc page is rendered
- **THEN** valid JSON-LD (`BreadcrumbList` + `TechArticle`) SHALL be present in the HTML

### Requirement: OG image for docs

A default docs OG image (1200x630, < 100KB) SHALL be used for all doc pages. Same dark background + Kaiord logo style as landing, with "Documentation" subtitle.

#### Scenario: Social sharing

- **WHEN** a doc page is shared on social media
- **THEN** the OG image SHALL render with the Kaiord branding

### Requirement: GitHub Releases link

The docs site footer SHALL include a link to GitHub Releases for project activity signal.

#### Scenario: Releases link

- **WHEN** a user views the docs footer
- **THEN** a "Releases" link SHALL navigate to the GitHub Releases page

### Requirement: Unified deployment

The deploy workflow SHALL build landing + editor + docs and merge into a single GitHub Pages artifact with `dist/docs/` containing the VitePress output.

#### Scenario: Docs included in deploy

- **WHEN** the deploy workflow runs
- **THEN** the merged artifact SHALL contain `docs/index.html` in addition to the existing landing and editor files
