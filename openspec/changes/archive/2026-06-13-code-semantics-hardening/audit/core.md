# Core Semantic Quality & Business-Language Audit — `@kaiord/core`

## Findings

### Dimension: model (domain modeling depth / primitive obsession)

- **[SEVERITY med] [model] `src/domain/schemas/krd/record.ts:30-38`** — every physiological quantity is a bare `z.number()`: `heartRate`, `cadence`, `power`, `speed`, `distance`, `verticalOscillation`, `stanceTime`, `stepLength`. Units live only in the _adapter_ convention (`speed` is m/s, `power` is watts) and in prose, not in the type. — Why it weakens legibility: a reader cannot tell from `record.speed: number` whether that is m/s, km/h, or pace-seconds; the unit is folklore. The codebase already proved it can reify units (target-values uses `{ unit: 'mps', value }` discriminated unions) but did NOT carry that discipline into the time-series record/session/lap layer, so the same concept (speed/power/HR) is modeled two different ways depending on file. — Improvement: at minimum add a one-line unit annotation per field (`/** watts */`, `/** m/s */`) as `krd/lap.ts` does for its _section_ comments; ideally introduce branded scalar types (`type Watts = number & {__watts}`) or reuse the target-value unit vocabulary so `record.speed` and `paceValue.mps` share one "meters-per-second" concept.

- **[SEVERITY med] [model] `src/domain/schemas/duration.ts:21-86`** — the field name carrying the magnitude _changes per variant_ (`seconds` | `meters` | `bpm` | `watts` | `calories`), and `repeat_until_*` variants reuse the same condition field (`bpm`, `watts`, `seconds`) as the _terminal_ variants with no marker distinguishing "this is a stop-condition threshold" from "this is the step's own duration." — Why it weakens legibility: `{ type: 'power_less_than', watts: 200 }` and `{ type: 'repeat_until_power_less_than', watts: 200, repeatFrom: 3 }` are semantically very different (one ends a step, one ends a _loop_), but the payload shape differs only by `repeatFrom`. The "repeat" family is really a separate domain concept (loop terminator) bolted onto the Duration union. — Improvement: this is _idiomatic-acceptable_ (discriminated union, illegal states largely unrepresentable), but a `repeatFrom`-bearing nested grouping (`{ kind: 'until', condition: {...}, repeatFrom }`) would make "this duration closes a repeat block" legible at the type level rather than by spotting an extra field.

- **[SEVERITY low] [model] `src/domain/schemas/workout-step.ts:18-24`** — `durationType`/`durationTypeSchema` and `targetType`/`targetTypeSchema` are stored redundantly alongside `duration.type`/`target.type`, then a `.refine` exists _only_ to assert they match (lines 29-36). — Why it weakens legibility: a denormalized discriminant duplicated for (per the comment elsewhere) FIT-adapter convenience is serialization concern leaking into the domain shape; the type system can't make the mismatch unrepresentable so a runtime refine is needed. — Improvement: acceptable as a documented round-trip affordance, but worth a comment on the schema saying _why_ the discriminant is duplicated (adapter ergonomics), since the refine alone reads as defensive paranoia rather than intent.

- **[SEVERITY low] [model] `src/domain/schemas/workout.ts:42-43`** — `poolLength: z.number()` + `poolLengthUnit: z.literal("meters")`. The unit is a one-value literal, so the field is pure ceremony, yet `convertLengthToMeters` (length-unit.converter.ts) accepts `"yards"`. — Why it weakens legibility: the converter speaks a two-unit vocabulary the schema forbids, so the domain says "pool length is always meters" while a sibling converter implies yards are a first-class input. Reader can't tell which is the truth. — Improvement: either widen `poolLengthUnit` to the `LengthUnit` enum the converter already uses, or drop the unit field and document that pool length is normalized-to-meters on ingest.

### Dimension: rules (business-rule legibility)

- **[SEVERITY med] [rules] `src/domain/schemas/health/sleep.ts:32`, `hrv.ts:10`, `weight.ts:12`, `stress.ts:11`, `daily.ts:14`, `body-composition.ts:15`** — the version gate `z.string().regex(/^2\.\d+$/)` is copy-pasted verbatim across all six health schemas (confirmed via grep). The business rule "health payloads are v2.x, additive minors allowed" is encoded as an anonymous regex in six places. — Why it weakens legibility: the rule has no name; a reader must reverse-engineer "2 dot one-or-more-digits" into "v2.x line" six times, and a future v3 bump means editing six literals. The test file even comments on the shared intent, proving the duplication is known. — Improvement: extract `export const HEALTH_SCHEMA_VERSION_PATTERN = /^2\.\d+$/;` (or a `healthVersionSchema` field schema) in `health/tolerances.ts`-style shared module and reference it. One named constant makes the rule a rule. **This is the single highest-leverage rules fix.**

- **[SEVERITY low] [rules] `src/domain/schemas/target-values/power.ts:31`, `heart-rate.ts:27`, `pace.ts:29`, `cadence.ts:27`** — physiological ceiling values (`5000` watts, `300` bpm, `30` m/s, `300` rpm, `1000` percent FTP) are inline magic numbers. The JSDoc explains the rationale well ("above any recorded human sprint peak"), so this is _mitigated_, but the numbers themselves aren't named. — Why it weakens legibility: the same `300` means "max bpm" in one file and "max rpm" in another; a shared `MAX_HUMAN_BPM = 300` would let the HR record clamp (`record.ts:30` also uses `max(300)`) and the HR target share one source of truth instead of two independent `300`s that happen to agree. — Improvement: promote the caps to named domain constants co-located with the health/physiology tolerances; the doc comment already supplies the name (`HUMAN_SPRINT_POWER_CEILING_WATTS`).

- **[SEVERITY low] [rules] `src/protocol/profile-snapshot.ts:18,105`** — `SNAPSHOT_LENGTH_LIMIT = 8192` and `STALE_SNAPSHOT_THRESHOLD_DAYS = 7` are _exemplary_ (named, JSDoc'd with rationale). Flagging the contrast: the FNV-1a seed/prime `0x811c9dc5` / `0x01000193` (lines 131-134) are unnamed magic. — Why it weakens legibility: minor — these are the canonical FNV-1a constants and the function JSDoc names the algorithm, so a knowledgeable reader recognizes them; but a `FNV_OFFSET_BASIS` / `FNV_PRIME` pair would make the hash self-documenting. — Improvement: name the two FNV constants.

### Dimension: abstraction (function semantics & abstraction level)

- **[SEVERITY low] [abstraction] `src/application/round-trip/validate-round-trip.ts:23-27` vs `119-158`** — the exported type `ValidateRoundTrip` and factory are _generic-sounding_ ("round trip") but the two methods are hard-wired to FIT: `validateFitToKrdToFit`, `validateKrdToFitToKrd`, and the doc says "between FIT and KRD formats." Yet the implementation only depends on injected `BinaryReader`/`BinaryWriter` — it is format-agnostic in mechanism but FIT-named in API. — Why it weakens legibility: a reader wanting to validate TCX↔KRD round-trips finds a use case whose name promises generality and whose method names deny it. The abstraction level of the _name_ (generic round-trip) doesn't match the _signature_ (FIT-specific). — Improvement: either rename methods to `validateBinaryRoundTrip` / `validateKrdRoundTrip` (honest about the injected port) or accept the FIT coupling and rename the export `validateFitRoundTrip`. The current half-and-half is the least legible option.

- **[SEVERITY low] [abstraction] `src/domain/schemas/duration.ts:1-6` and `target.ts:12-13,52-60`** — these files mix schema definition with re-export plumbing (`export type { DurationType } from "./duration-type"`). Minor, idiomatic barrel behavior — noted only because it slightly dilutes the "this file defines the Duration concept" reading. No action needed.

- **EXEMPLARY [abstraction] `src/application/round-trip/compare-entity-fields.ts` + `check-field.ts`** — the table-driven `RECORD_CHECKS`/`LAP_CHECKS`/`SESSION_CHECKS` design (e.g. `compare-records.ts:8-14`) reads as a _specification_ of which fields are tolerance-compared and with which checker. `compareEntityFields` stays at one abstraction level and its JSDoc states the business rule ("only flags values that drifted, never values that were absent"). This is the pattern to replicate.

### Dimension: boundary (ports & boundaries)

- **[SEVERITY med] [boundary] `src/ports/auth-provider.ts:11-13` and `src/ports/workout-service.ts:10-11`** — port interfaces use snake*case (`is_authenticated`, `export_tokens`, `restore_tokens`, `created_at`, `updated_at`) inside an otherwise camelCase TypeScript codebase. — Why it weakens legibility: per CLAUDE.md, domain schemas are snake_case and \_adapter* schemas are camelCase; ports are domain contracts and should speak camelCase domain language, but these leak the Garmin SSO wire convention (`is_authenticated`) into the port vocabulary. The `ports/AGENTS.md` _acknowledges_ this ("preserve that asymmetry … for historical Garmin SSO compatibility"), so it's a _deliberate, documented_ exception — but it still means a port consumer must know which methods are wire-shaped and which aren't. — Improvement: since it's a documented historical constraint, low urgency; if ever revisited, give the port camelCase names (`isAuthenticated`, `exportTokens`) and let the _garmin-connect adapter_ map to the SSO wire shape — that's exactly where serialization vocabulary belongs.

- **[SEVERITY low] [boundary] `src/ports/workout-service.ts:9` & `auth-provider.ts:4`** — `WorkoutSummary.sport: string` (untyped, not `sportSchema`) and `TokenData = Record<string, unknown>`. The `sport: string` is consistent with `krdSession.sport` (documented forward-compat), so acceptable. `TokenData` as opaque bag is correct boundary hygiene (the port deliberately doesn't know token internals). No action.

- **EXEMPLARY [boundary] `src/ports/format-strategy.ts`** — `BinaryReader`/`TextReader`/`BinaryWriter`/`TextWriter` are pure domain-language port aliases (`(buffer) => Promise<KRD>`) with zero SDK leakage. The whole strategy contract is four lines and reads as the conversion vocabulary itself. Replicate.

### Dimension: errors (error & log language)

- **[SEVERITY low] [errors] `src/domain/validation/extract-workout.ts:28-31`** — `"Missing or invalid extensions.structured_workout"` surfaces the internal JSON _path_ (`extensions.structured_workout`) to whoever catches the error. — Why it weakens legibility: an integrator who provided a KRD without a workout body gets jargon about an extensions namespace rather than "This KRD does not contain a structured workout." For a _validation boundary for agent-provided data_ (per the converter's own JSDoc), the message should be domain-phrased. — Improvement: lead with the business statement ("KRD is not a structured workout: …") and keep the path in the structured `field` for machine consumers (it's already there).

- **EXEMPLARY [errors] `src/domain/schemas/health/sleep.ts:50-55` and `src/domain/zones/power-zones.ts:53,82-84`** — the sleep stage-sum message ("Sum of stage durations (Ns) diverges from totalDurationSeconds by Ds, exceeding the ±60s tolerance") and the zone errors ("No canonical power zone matches X% FTP. Expected one of: 55, 75, …") are _outstanding_: they state the invariant, the actual values, and the allowed set. This is the gold standard for the whole codebase.

- **EXEMPLARY [errors] `src/domain/schemas/krd/index.ts:107-118`** — the conditional `metadata.sport` refinement messages name the KRD type and the rule ("KRD type X requires metadata.sport to be a non-empty string" / "must omit metadata.sport (health records have no associated sport)"). Domain-legible.

## Three Highest-Leverage Improvements

1. **Name the health version gate once** (`src/domain/schemas/health/*.ts`). Replace six copies of `z.string().regex(/^2\.\d+$/)` with a single named `HEALTH_SCHEMA_VERSION_PATTERN` / shared field schema. Converts an anonymous, duplicated regex into a named business rule and makes the inevitable v3 bump a one-line change. Lowest effort, highest rules-legibility gain.

2. **Unify the unit vocabulary across the time-series layer** (`src/domain/schemas/krd/record.ts`, `session.ts`, `lap.ts`). At minimum annotate every physiological field with its unit (matching the section-comment discipline `lap.ts` already shows); ideally introduce branded scalar types or reuse the target-value unit concept so `record.speed` and `paceValue.mps` are recognizably the same "m/s." Closes the gap between the strong target-values modeling and the weak record modeling.

3. **Make the round-trip use case honest about its abstraction level** (`src/application/round-trip/validate-round-trip.ts`). Either rename the methods to reflect the injected ports (`validateBinaryRoundTrip`) or rename the export to admit the FIT coupling (`validateFitRoundTrip`). Remove the name-vs-signature contradiction so the API reads as what it does.

## Verdict

**Does core speak the business language? Mostly yes — fluently in its newer and most load-bearing modules, with a legacy accent in a few corners.** The target-values discriminated unions, the health-record invariants with their excellent error messages, the power-zone domain table, the table-driven round-trip comparators, and the format-strategy ports all read like fitness-domain prose: units are reified, illegal states are largely unrepresentable, and rules announce themselves with named constants and value-rich messages. The weak spots are concentrated and consistent: the time-series record/session/lap schemas fall back to bare `number` with units carried only by folklore (a visible regression from the target-values layer's discipline), one core business rule (the v2.x health version gate) is duplicated as an anonymous regex six times, and a handful of port methods carry snake_case wire vocabulary that the codebase itself documents as a tolerated historical exception. None of these obscure correctness — they cost a reader the ability to learn the domain from the types alone in those specific files. Tightening the three items above would move core from "speaks the business language well" to "speaks it uniformly."
