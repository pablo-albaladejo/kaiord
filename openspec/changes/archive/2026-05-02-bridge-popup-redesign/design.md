## Context

Two browser extensions ship today:

- `@kaiord/garmin-bridge` — proxies SPA requests to Garmin Connect via the user's session.
- `@kaiord/train2go-bridge` — proxies SPA requests to Train2Go via the user's session.

Both:

- Share the **identical PNG icon set** (`icons/icon{16,48,128}.png`) — a dark navy hexagon with a blue inner mark. There is no SVG master in the repo. With both pinned to the toolbar, users cannot visually distinguish them.
- Render a popup whose primary content is a status pill plus one debug action button (`List Workouts` for Garmin, `Read This Week` for T2G). The action data is not retained, the popup is not athlete-aware, and the buttons feel like leftover developer affordances rather than user features.
- Speak to the SPA over `externally_connectable` using a small action vocabulary (`ping`, `list`, `push`, `read-week`, `read-day`, `open-*`). The bridges hold no athlete state; the SPA's Dexie store is the sole source of truth for the active profile, sport zones, FTP, body weight, threshold pace, etc.
- Are governed by `openspec/specs/garmin-bridge`, `openspec/specs/train2go-bridge`, and the cross-cutting `openspec/specs/spa-bridge-protocol`. The Train2Go spec already has a "Popup UI" requirement that mentions `Read This Week`; the Garmin spec does not specify popup contents.

Constraints worth surfacing up front:

- **Privacy / CWS**: any change that adds new outbound URLs, new content-script allowlist entries, or new `host_permissions` triggers a Chrome Web Store privacy-justification review (see `openspec/specs/cws-train2go-listing` and `openspec/specs/extension-store-publish`). This change deliberately stays inside the existing surface.
- **Hexagonal layering** (`/Users/pablo/.claude/CLAUDE.md`): domain (SPA profile + zones) cannot reach into infrastructure (browser-extension storage). Communication must go through the existing bridge port; bridges cannot acquire domain logic.
- **Mechanical guards over AI review** (user preference): icon recolor, snapshot schema, and popup contract must be validated by deterministic checks (build script + Zod schema + Vitest), not eyeballing.

## Goals / Non-Goals

**Goals:**

- Two icons that are unambiguously distinguishable in the browser toolbar at 16×16 while preserving the Kaiord silhouette so the family relationship is obvious.
- A single SVG master that produces both color variants × three PNG sizes, regenerated by a `pnpm icons:build` script. The script is co-located with a `*.test.mjs` per `scripts/` convention.
- Popup content that prioritizes "who am I as an athlete on this platform, and is the connection alive" over debug actions.
- A typed `profile-snapshot` action on the SPA↔Bridge protocol that lets the SPA push athlete data to a bridge's `chrome.storage.local`. The bridge surfaces that data; it does not interpret it.
- Auto-fetch on popup open. Failure paths surface a `Retry` affordance; success paths do not show a button.
- No CWS surface change: `host_permissions`, content-script allowlists, and outbound URLs are byte-identical before and after this change.

**Non-Goals:**

- Adding upstream profile pulls (e.g., `/userprofile-service/userprofile/personal-information` on Garmin). That is an explicit _future_ change and is excluded here so we can ship without re-litigating the privacy review.
- Showing full per-zone tables in the popup. The popup is a 340 px viewport; zone editing remains in the SPA.
- Allowing the bridge to mutate, validate, or transform the snapshot. Bridges are dumb pipes; they store and surface the bytes the SPA sent.
- Replacing the SPA's bridge registry. The new action plugs into the existing `BridgePort`.

## Decisions

### D1 — Icon master is a single SVG with a color token; PNGs are derived

**Decisions (committed):**

1. One shared SVG master at `packages/_shared/extension-icon/master.svg` consumed by `scripts/build-extension-icons.mjs` with the accent color passed as a CLI flag (`--bridge garmin|train2go --accent <hex>`). No per-bridge masters. The script renders three sizes (16, 48, 128) by substituting the accent and rasterizing.
2. **Rasterizer:** `sharp` as a repo-root devDependency (script-only, NOT bundled into either extension). If `pnpm why sharp` at implementation time reveals it is unsuitable on the target CI runners (e.g., Apple Silicon native binary issues that have not been resolved), the documented fallback is `@napi-rs/canvas`. The decision is sequenced (verified at apply-time in tasks 1.2) but not architecturally open — both choices satisfy "JS-friendly raster of a single SVG via a Node script in `scripts/`".

**Why over alternatives:**

- _Per-bridge hand-edited PNGs (status quo)_: drift between sizes and between bridges. Already produced two visually identical icons by accident.
- _Per-bridge SVG masters (no shared source)_: duplication, recolor must be done twice, future Kaiord-family extensions duplicate again.
- _Runtime CSS recolor inside the popup_: only recolors the popup-internal icon, not the toolbar icon. Toolbar icons are sourced from the static PNGs in the manifest; only build-time rasterization works.

**Hexagonal layer:** infrastructure / build tooling. No domain impact.

**Mechanical guard hardening:** the inter-icon mean-color-delta check is necessary but insufficient — a thin accent stripe could pass it. Pair it with a _pixel-mass_ check: at 16×16, the accent color (within ±15° hue tolerance of the bridge's accent token) SHALL constitute ≥25% of non-transparent pixels. A 2px-stripe accent fails this; a hex-fill accent passes.

**Justification for `sharp`**: already a transitive dep in the workspace via several tooling packages; if not, alternative is `@napi-rs/canvas`. `sharp` is preferred — smaller, no native build complications on macOS/Linux runners. Confirm via `pnpm why sharp` before adding as a direct dep. The rasterizer ships only as a repo-root devDependency consumed by the build script; it is **not** bundled into either extension.

### D2 — Athlete data flows SPA → Bridge via a new `profile-snapshot` action; bridges store, never interpret

The SPA pushes a profile snapshot to each registered bridge:

- Trigger 1: any change to the active profile in Dexie (`useLiveQuery` effect).
- Trigger 2: bridge transitions to VERIFIED (heartbeat success after a previous UNAVAILABLE / fresh registration).

Bridge background:

```
chrome.runtime.onMessageExternal.addListener((msg, sender, sendResponse) => {
  if (msg.action === "profile-snapshot") {
    chrome.storage.local.set({
      profileSnapshot: { ...msg.snapshot, receivedAt: Date.now() }
    }, () => sendResponse({ ok: true, protocolVersion: 1, data: null }));
    return true;
  }
  // ... existing actions
});
```

Bridge popup reads `chrome.storage.local.get("profileSnapshot")` on open. If present, render the athlete card; if absent or older than 7 days, render a placeholder ("Open the Kaiord editor to populate this card") and a deep link to the SPA.

**Why over alternatives:**

- _Bridge fetches profile from upstream_: requires CWS allowlist expansion (Garmin) or doesn't exist (T2G doesn't expose zones). Out of scope per Non-Goals.
- _Bridge reads SPA's IndexedDB directly_: cross-origin IndexedDB access from extension popup is not possible without a content script on the SPA origin and additional message passing — strictly more complex than just sending a snapshot.
- _Bridge embeds a tiny SQLite/Dexie of its own_: duplicates state, requires sync logic, violates "bridges are dumb pipes".

**Hexagonal layer:** SPA application layer pushes; bridge infrastructure stores. The snapshot DTO is owned by `spa-bridge-protocol` (a cross-cutting capability spec), not by either bridge.

**Origin gate:** the snapshot is sent to whichever origin is permitted by the manifests' `externally_connectable.matches` (kaiord.com + localhost in dev). There is no per-action origin allowlist tighter than the manifest. The privacy-surface guard's byte-equality check on `externally_connectable.matches` IS the origin gate for snapshot data; widening it requires a deliberate golden-snapshot update reviewed in the diff. As defense-in-depth, the bridge action handlers ALSO verify `sender.origin` against the manifest's origin set and reject with `{ ok: false, error: "Origin not permitted", retryable: false }` if it does not match — even though Chrome already filters at the platform level.

### D3 — Snapshot schema is small, sport-keyed, and Zod-validated on both ends

The schema lives in `@kaiord/core` (`packages/core/src/types/profile-snapshot.ts`), not in the SPA package. Reason: the snapshot is the cross-cutting protocol contract between the SPA and any present or future bridge. Hosting it in the SPA package would make every future bridge transitively depend on the SPA editor — a layering inversion. `@kaiord/core` is the natural home for cross-package contracts and already ships pure Zod schemas. (`@kaiord/core` already hosts the `bridgeManifestSchema` for the same reason — it is the only package both the SPA and any bridge can structurally agree on without a layering inversion. The `ProfileSnapshot` follows the same precedent.)

```ts
// packages/core/src/types/profile-snapshot.ts
const profileSnapshotSchema = z.object({
  schemaVersion: z.literal(1),
  profile: z.object({
    name: z.string().min(1).max(100),
    bodyWeight: z.number().positive().optional(),
  }),
  activeSport: z.enum(["cycling", "running", "swimming"]).optional(),
  thresholds: z
    .object({
      cycling: z
        .object({ ftp: z.number().int().positive().optional() })
        .optional(),
      running: z
        .object({
          thresholdPaceSecPerKm: z.number().int().positive().optional(),
          lthr: z.number().int().positive().optional(),
        })
        .optional(),
      swimming: z
        .object({
          cssPaceSecPer100m: z.number().int().positive().optional(),
        })
        .optional(),
    })
    .default({}),
  heartRate: z
    .object({
      max: z.number().int().positive().optional(),
      lthr: z.number().int().positive().optional(),
    })
    .default({}),
  generatedAt: z.iso.datetime(),
});
```

Notes:

- `schemaVersion` is explicit so the bridge can refuse incompatible payloads cleanly (return `{ ok: false, error: "Unsupported snapshot schema version", retryable: false }`).
- Zone tables are intentionally NOT included — the popup shows summary numbers, not per-zone rows. Smaller payload, smaller schema surface.
- **Bridge validator strategy (committed):** _duplicated_. Each bridge ships a plain-JS structural validator (`packages/{garmin,train2go}-bridge/profile-snapshot.js`) that mirrors the Zod schema's shape rules. A parity test loads a shared fixture set (positive + negative cases) from `@kaiord/core/test-utils` and asserts that the SPA Zod parser and each bridge's plain-JS validator agree on every fixture. Vendoring is rejected because the bridges have no module bundler; a manual sync step kept honest by a parity test is the simplest path. **At runtime the bridges have NO dependency on `@kaiord/core`** — the runtime validator is hand-rolled plain JS in the bridge package. The dependency on `@kaiord/core/test-utils` is exclusively a `devDependency` for the parity test; each bridge's `package.json` MUST keep `@kaiord/core` out of `dependencies`.
- **Canonical serialization for byte budget:** the 8 KiB cap applies to `JSON.stringify(snapshot)` of the post-Zod-parse object on the SPA side AND to `JSON.stringify(input)` of the validator's input on the bridge side. Both ends use Node/Browser native `JSON.stringify` with no replacer or spacer, so output is deterministic for a given own-key insertion order. The SPA's mapper SHALL build the snapshot with the field order documented in the DTO (insertion-order matches schema definition order); a unit test pins the canonical byte length of the baseline fixture so the limit is reproducible.
- **Prototype-pollution defense:** the plain-JS validator MUST construct outputs via `Object.create(null)`, MUST reject inputs whose own keys include `__proto__`, `constructor`, or `prototype` (using `Object.getOwnPropertyNames` to bypass any inherited shadowing), and MUST NOT use `Object.assign({}, input)` or spread (`{...input}`) on attacker-controlled payloads. Inputs failing this check are rejected with `{ ok: false, error: "Invalid snapshot payload", retryable: false }` and the global `Object.prototype` SHALL be unmodified.
- **Snapshot serialized-length budget:** the validator rejects any payload whose `JSON.stringify(snapshot).length` exceeds 8192 UTF-16 code units (NOT bytes — see clarification below). `chrome.storage.local` has a 10 MB total quota, but a malicious or malformed snapshot SHALL NOT be allowed to exhaust it. Practical snapshots are <2000 code units. Note: 8192 UTF-16 code units is up to ~32 KiB on disk in the worst case (all-non-BMP characters), still <0.4% of the 10 MB quota — so the cap is generous on the storage side and tight on the validator side. The shorthand "8 KiB cap" used elsewhere in this design refers to this 8192-code-unit limit.
- **Content fingerprint for de-duplication:** `@kaiord/core` exports a pure `fingerprintSnapshot(snapshot): string` that returns a stable hash (e.g., FNV-1a 32-bit hex of canonical JSON) of `(profile.id, snapshot fields excluding generatedAt)`. The SPA's push adapter persists `lastSuccessfulFingerprint[bridgeId]` and skips a push whose fingerprint matches. Excluding `generatedAt` makes the de-dup robust to clock skew, system sleep, and SW restart — two snapshots that differ ONLY because they were derived seconds apart from identical state collapse to one transport call. The function is unit-tested in `@kaiord/core` and reused by the SPA push adapter; bridges do NOT depend on it (de-dup is a SPA-side optimization, the bridge cache write is idempotent regardless).
- **Future versioning:** the v2 schema (whenever required) SHALL be a discriminated union: `z.discriminatedUnion("schemaVersion", [v1Schema, v2Schema])`. Bridges accept any version they were built with; the SPA pushes the highest version a bridge advertises in its manifest (`capabilities` array gains a `snapshot:v2` token when v2 ships, and the SPA falls back to v1 for older bridges). The current `z.literal(1)` is the v1 entry of that future union.

**Hexagonal layer:** schema lives in `@kaiord/core` (cross-cutting domain contract). The SPA derives the snapshot from its domain `Profile` (a pure mapper). The bridge holds a structural copy purely for input validation at its boundary — this matches the existing pattern (`bridgeManifestSchema` is duplicated SPA-side and parity-tested).

### D4 — Auto-fetch on popup open with bounded latency; `Check Session` is a Retry affordance

`popup.js` runs the data-load pipeline immediately:

```
loadPopupData() :=
  ping  ⨯  storage.get(profileSnapshot)  ⨯  per-bridge rollup
   │              │                              │
   │              │                              ├─ T2G: read-week(today, userId)
   │              │                              │      (cached for 5 min in chrome.storage.local
   │              │                              │       under "lastWeeklyRollup")
   │              │                              └─ GC:  workouts list metadata already-done in ping
   │              │
   │              └─ render athlete card
   │
   └─ render status pill
```

**Latency contract:** the auto-fetch pipeline phases have differentiated wall-clock timeouts:

- Snapshot read (`chrome.storage.local.get`): **1 s** (local, should be near-instant).
- Ping (`{ action: "ping" }`): **3 s** (one round-trip through the content script).
- Rollup phase: **3 s** for Garmin (workout-library count is already in the ping envelope, no extra fetch); **8 s** for Train2Go (one `read-week` call returning ~3 weeks of HTML, parsed in the SW). The 8 s budget is empirically generous on a healthy session and avoids false-positive disconnected states on slow upstreams.

On timeout of the **ping** or **snapshot** phase, the popup renders the **disconnected state** with a single `Retry` button. On timeout of the **rollup** phase ONLY (i.e., ping succeeded), the popup renders the **connected state** with the rollup line replaced by "Rollup unavailable — try again". This prevents conflating a slow weekly fetch with an expired session. The in-flight popup-side await is aborted via `AbortController` in all timeout cases. The popup ALSO wires `window.addEventListener("pagehide", () => controller.abort())` so any awaits in flight when the user closes the popup are released.

**Important scope of the abort:** the `AbortController` cancels the _popup-side promise_ that awaits `chrome.runtime.sendMessage`'s callback. It does **NOT** cancel the underlying network call running in the bridge service worker. The SW continues the `garminFetch`/`train2goFetch` until the existing 30-second content-script timeout fires (or the response arrives, whichever comes first). The popup-side abort guards UI rendering and prevents zombie popups from acting on resolved-after-close responses; it does not reduce network or CPU cost in the SW. This is acceptable because the SW work is already capped by the existing 30 s allowlist timeout, and the cost is bounded.

**T2G rollup TTL:** `read-week` returns ~3 weeks of HTML (~200 KiB), parsed in the bridge service worker. To avoid burning that on every popup open, the popup caches the parsed rollup `{ planned, done, workload }` in `chrome.storage.local` under `lastWeeklyRollup` with a 5-minute TTL. On cache hit, no `read-week` call. On cache miss, fetch + write-through.

**Concurrent cache-miss behavior:** two popup opens within the TTL window both observing a miss is acceptable noise. We do NOT introduce an in-memory promise cache in the SW to coalesce calls — popups are infrequent (user-initiated), `read-week` is idempotent, and the last write to `lastWeeklyRollup` wins. A spec scenario explicitly accepts the duplicate-fetch behavior so a future implementer does not over-engineer it.

**Updated relative time:** the "Updated <relative time>" line uses `Intl.RelativeTimeFormat("en", { numeric: "auto" })`, snapping the delta to the largest unit ≥ the threshold:

| `now - receivedAt` | Rendered text                                   |
| ------------------ | ----------------------------------------------- |
| < 60 s             | "Updated just now"                              |
| < 60 min           | "Updated <N> minutes ago"                       |
| < 24 h             | "Updated <N> hours ago"                         |
| < 7 d              | "Updated <N> days ago"                          |
| ≥ 7 d              | replaced by stale-snapshot placeholder (see D6) |

**Staleness threshold (committed):** 7 days, exposed as `STALE_SNAPSHOT_THRESHOLD_DAYS = 7` in `@kaiord/core` (alongside the schema). Spec scenarios reference the constant. Rationale: training-week cadence; revisit if telemetry from registered bridges suggests otherwise.

**Localization (out of scope):** the popup is English-only in v1. `Intl.RelativeTimeFormat` is invoked with the `"en"` locale literal; the placeholder copy ("No profile yet…", "Profile has no thresholds yet…", "Rollup unavailable — try again") is hard-coded English. If telemetry shows non-English usage, revisit by passing `navigator.language` to `Intl.RelativeTimeFormat` and externalising the strings.

**Rendering rules:**

- If `ping` fails or returns `sessionActive=false`, the popup renders the disconnected card AND a single `Retry` button.
- On the success path: no "Check Session" or "Retry" button. A small refresh-icon button (`⟳`, 16px, accent color, `aria-label="Refresh"`) lives in the popup header for power-users who want to force a re-fetch (it bypasses the rollup TTL).
- The athlete card always shows a relative-time line ("Updated <Nd> ago") with `aria-live="polite"` so when stale-snapshot replacement happens, the user understands why.

Rationale: the current popup forces the user to click `Check Session` to populate the status pill, which they then have to read. Auto-fetch removes a click for the common case and reserves the prominent button for the error case where the user has agency (re-login, refresh). The header refresh-icon recovers the power-user resync affordance without dominating the layout.

### D5 — Coach name (T2G) and display name (Garmin) come only from data already on the wire

- T2G `parsePingJson` gains an opportunistic field extractor for coach/trainer name from `data.user.coach` (or whatever the live payload exposes — verify on the wire as a sub-task before relying on it). The action's allowed path stays exactly `^/api/v2/profile/ping$`.
- Garmin: the existing `/workout-service/workouts?start=0&limit=1` response envelope is inspected for any user-display fields. If absent, the field is simply not rendered. **No new path is added to the content-script allowlist**, even if it would be technically simple.

This keeps the change CWS-neutral and respects user preference for mechanical guards (the test for the Garmin content-script allowlist must remain unchanged).

### D6 — Popup info-architecture is identical across both bridges; only color and per-bridge rollup differ

Shared layout:

```
[ accent dot ]  Bridge name                            [⟳]
─────────────────────────────────────────────────────────
[●] Connected as <user>                                    ← status pill (color + glyph)
    <upstream-only sub-line>                               ← coach (T2G) / display name (GC)
─────────────────────────────────────────────────────────
ATHLETE                                                    ← from profile-snapshot
  FTP, threshold pace, LTHR, max HR, sport
  Updated <N hours/days> ago                               ← aria-live="polite"
─────────────────────────────────────────────────────────
THIS WEEK / SYNC                                           ← per-bridge rollup
─────────────────────────────────────────────────────────
[ Open editor ]                                            ← primary CTA, full-width
Open <upstream> ↗                                          ← secondary text link
```

**Layout tokens (committed):**

- Popup viewport: 340 px wide, auto height, 16 px padding.
- Typography: `system-ui, sans-serif`. Title 14 px / 600. Body 13 px / 400. Captions ("ATHLETE", "THIS WEEK") 11 px / 700 uppercase letter-spacing 0.04em.
- Accent dot: 8 px circle, accent color, 8 px right margin from title.
- Status pill: 8 px padding, 4 px border-radius, leading 14 px glyph (✓ for connected, ✗ for disconnected) + text label.
- Athlete card: 2-column micro-grid (`grid-template-columns: 1fr 1fr; gap: 4px 12px`). Each cell renders one (label, value) pair to avoid the 60-char wrap problem at 13 px.
- Refresh-icon: 16 px ⟳ glyph, accent color, `aria-label="Refresh"`. Lives in a **reserved 16×16 slot** at the top-right of the header (header is a 3-column grid: `[accent-dot 8px] [title 1fr] [refresh-slot 16px]`). During loading and on the failure path the icon uses `visibility: hidden` (NOT `display: none`) so the slot remains reserved and the title row does not reflow.
- Athlete card field-count rendering: 0 fields with snapshot present → "Profile has no thresholds yet" placeholder (distinct from the no-snapshot copy); exactly 1 field → render full-width single cell; ≥2 fields → render the 2-column grid as above.

**Accessibility tokens:** decorative glyphs (the leading ✓/✗ in the status pill, the ⟳ refresh icon, accent-dot) SHALL be `aria-hidden="true"` so screen readers do not double-announce alongside the `aria-label`/visible text. Font sizes are in `px` (not `rem`) because Chrome popup rendering has its own context independent of the host page's root font size — `rem` would be unpredictable. The loading-skeleton shimmer animation SHALL be suppressed under `@media (prefers-reduced-motion: reduce)`.

- Primary CTA: full-width button, 8 px vertical padding, accent background, white text. Secondary link: text-only, accent color, underlined, 12 px height.

**Shared assets package:** `packages/_shared/` is a new monorepo top-level for non-publishable assets shared across packages (the SVG icon master, the popup CSS master). It SHALL be configured as `"private": true` in its own `package.json`, included in `pnpm-workspace.yaml`'s `packages:` glob, and documented at the top of its `README.md`. It is NOT published to npm and has no version. If a future asset has runtime code (not just static files), promote it to a published `@kaiord/internal-*` package instead.

**Shared CSS:** the structural CSS (everything except the accent and accent-hover color tokens) lives at `packages/_shared/popup/popup.css` and is copied byte-for-byte into both `packages/{garmin,train2go}-bridge/popup.css` by a small `pnpm popup:sync` script. A mechanical guard (`scripts/check-popup-css-parity.test.mjs`) asserts both copies are byte-identical to the master. Color tokens (`--accent`, `--accent-hover`) are the only per-bridge divergence and are defined in each bridge's `popup.html` `<style>` block.

**Open editor CTA target:** the "Open editor" primary CTA opens the production SPA URL `https://app.kaiord.com/`. The dev variants of the bridges (loaded unpacked from `manifest.json`, not `manifest.prod.json`) open `http://localhost:5173/` instead. The DTO does NOT carry a `spaUrl` field — the popup picks the URL from a build-time constant (`OPEN_EDITOR_URL`) baked into each bridge's `popup.js` based on whether it was built from the dev or prod manifest. Rationale: the SPA URL is environment-determined, not profile-determined, so embedding it in the snapshot would be a layering smell.

### D7 — Tests and mechanical guards

- **Icon build** (`scripts/build-extension-icons.test.mjs`): produces six output files with non-empty bytes and the expected dimensions; asserts the bridges' icons differ at every size (mean pixel-color delta above a threshold) AND that the accent color occupies ≥25% of non-transparent pixel mass at 16×16 (within ±15° hue tolerance of the bridge's accent token). Fails the lint job if drift returns.
- **Snapshot schema** (`packages/core`): Zod schema unit tests (parses valid payloads, rejects malformed, rejects unknown `schemaVersion`, rejects payloads larger than 8 KiB). Fixture set is exported from `@kaiord/core/test-utils` for parity tests.
- **Bridge structural validator** (`packages/{garmin,train2go}-bridge/test/profile-snapshot.test.js`): plain-JS parity tests against the same fixture set. Includes a _prototype-pollution_ fixture (`{ __proto__: { isAdmin: true }, schemaVersion: 1, ... }`) that MUST be rejected; post-test global `Object.prototype` SHALL be unmodified.
- **Popup data-load** (`packages/{garmin,train2go}-bridge/test/popup.test.js`): jsdom + chrome stubs; asserts auto-fetch, snapshot rendering, retry-only-on-failure, stale-snapshot placeholder, 3-second timeout fallback to disconnected state, refresh-icon affordance triggers re-fetch, ARIA attributes present, Retry button auto-focuses on failure, T2G rollup cache hit skips `read-week`.
- **Privacy-surface guard** (`scripts/check-bridge-privacy-surface.mjs` + `*.test.mjs`): byte-equality check on `host_permissions`, `content_scripts.matches`, `externally_connectable.matches`, and the content-script `ALLOWED` exports against a checked-in golden snapshot. Also greps `popup.js` for `fetch(`/`XMLHttpRequest` calls and asserts every URL is a relative path (no `http(s)://` scheme) — no exfil window introduced by popup code.
- **Manifest permission check**: a script-level test asserts both bridges declare `storage` in `permissions` (Garmin already does; Train2Go gains it as part of this change). Catches accidental drops in the prod manifest.
- **Spec lint**: every new requirement in the deltas has at least one `#### Scenario:` block, enforced by `pnpm lint:specs`.

## Risks / Trade-offs

- **Stale snapshot at first install** → Mitigation: popup renders an explicit "No profile yet. Open Kaiord to set FTP, pace, and HR." placeholder with a deep link; not a generic empty card. Snapshot is also pushed automatically once the SPA detects the bridge becoming VERIFIED, so the staleness window collapses to "user hasn't opened the SPA yet". Acceptable — this is a launcher tool, not a primary surface.
- **Snapshot schema drift between SPA and bridge** → Mitigation: `schemaVersion` field, Zod-on-SPA + plain-JS validator-on-bridge with a parity test that loads a shared fixture set, explicit reject path with a non-retryable error. Versions are bumped in lockstep via the changeset.
- **Concurrent snapshot pushes** (two SPA tabs editing different fields) → Mitigation: last-write-wins is acceptable because the snapshot is derived from the SPA's persisted active profile (a shared single source of truth). Both tabs will derive identical snapshots after the persistence layer commits. Both pushes ack `ok: true`; the bridge cache reflects whichever arrived second. Specified explicitly.
- **Snapshot push storms under bridge flap** → Mitigation: SPA-side de-duplication. If the next push to a bridge would have the same `(profile.id, generatedAt-truncated-to-second)` as the most recent successful push, skip it without charging the rate limiter. Specified explicitly.
- **`chrome.storage.local` is unencrypted on disk** → Mitigation: documented as a known constraint. The data class (athlete PII: name, body weight, FTP, HR data) is comparable to what is already in the SPA's IndexedDB. A local-machine attacker with browser-profile-directory access can already read the SPA's data; the bridge cache adds no new attack surface. This is consistent with the existing `chrome.storage.session` CSRF-token model (which uses session storage because the token IS a credential). Snapshot data is not credentials, so `local` is appropriate. Note that `profile-snapshot-clear` removes the entry from the `chrome.storage.local` API surface but does not guarantee on-disk physical erasure — Chrome backs `storage.local` with LevelDB, whose compaction is non-deterministic and may retain tombstoned entries until a future compaction pass. For higher assurance, `chrome.storage.session` (memory-only) would be required, but `session` doesn't survive SW restart, defeating the popup-on-cold-open purpose. The `local` choice is documented and accepted; users requiring stronger erasure can uninstall the extension (which triggers a Chrome-managed wipe of the extension's storage namespace).
- **Prototype-pollution via `profile-snapshot`** → Mitigation: plain-JS validator constructs outputs with `Object.create(null)`, rejects own-keys `__proto__`/`constructor`/`prototype` using `Object.getOwnPropertyNames`, never spreads attacker input. A negative-fixture parity test asserts `Object.prototype` is unmodified after a poisoning payload.
- **Snapshot exceeds storage quota** → Mitigation: 8 KiB validator cap on JSON-serialized payload. Larger inputs rejected with `retryable: false`.
- **Right-to-be-forgotten on profile delete** → Mitigation: complementary `profile-snapshot-clear` action. SPA pushes it when the active profile is deleted; bridges remove the storage key. Popup falls back to placeholder.
- **Icon recolor doesn't actually distinguish at 16×16** → Mitigation: build-script test computes inter-icon mean pixel delta AND requires the accent color to occupy ≥25% of non-transparent pixel mass at 16×16. A thin-stripe accent that passed the mean-delta check would fail the mass check.
- **`sharp` not present transitively or fragile on CI** → Mitigation: confirm via `pnpm why sharp` before committing to it. Fallback options: `@napi-rs/canvas` (pure-JS-ish), or pre-rendering once locally and committing the PNGs while keeping the script as the maintenance path. The rasterizer ships only as a repo-root devDependency consumed by the build script; it is never bundled into the published extensions.
- **Coach name field changes upstream** → Mitigation: extractor returns `undefined` on miss; popup hides the line. No hard dependency, no schema break.
- **`coachName` leakage in SPA logs** → Mitigation: documentation note that `response.data` raw payloads MUST NOT be logged in the SPA. The registry path uses Zod stripping and is unaffected.
- **Privacy regression** → Mitigation: a CI test asserts the _strings_ of `host_permissions`, `content_scripts.matches`, `externally_connectable.matches`, the content-script `ALLOWED` arrays, AND popup.js `fetch(`/`XMLHttpRequest` call sites (must be relative URLs only) are byte-identical to a checked-in golden snapshot. Any future PR that drifts these surfaces or introduces an absolute-URL fetch from popup code lights up the lint job.
- **Unaccounted CWS metadata edit** → Mitigation: the only metadata change is Train2Go gaining `storage`. Documented in the proposal Impact section and reflected in the privacy-surface golden update — the golden update is the only file in this PR that touches the surface, and it is reviewed in the diff.

## Migration Plan

1. **Roll out the snapshot push from the SPA first.** Old bridges (pre-action) will respond with `Unknown action: profile-snapshot` — handled gracefully (the SPA logs and continues; no user impact).
2. **Ship bridge updates** with the new popup + action handler. The Chrome Web Store auto-update propagates over 24–48 h.
3. **Telemetry / verification**: post-rollout, the `bridge-runtime-discovery` registry surfaces each bridge's version. After T+72 h, fewer than 1% of registered bridges should remain on the pre-redesign version. (No dashboards to wire — this is informational; the in-SPA version display already exists.)
4. **No rollback story needed** beyond shipping a CWS update with the previous build artifact. The change is additive on the protocol side; reverting bridges to v7.x leaves the SPA's snapshot push as a no-op.
