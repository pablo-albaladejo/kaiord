> Completed: 2026-05-25

## Why

Garmin FIT files routinely carry health-domain data — sleep stages, body weight, HRV summaries, daily wellness (steps / active calories / intensity minutes), body composition, and stress episodes — emitted by Garmin watches and connected accessories alongside workout activities. The Kaiord ecosystem is currently 100% workout-centric across every layer: the KRD canonical format enumerates only `structured_workout | recorded_activity | course`, the FIT adapter silently drops every message whose number is not registered in `FIT_MESSAGE_NUMBERS` (today only `FILE_ID(0)`, `WORKOUT(26)`, `WORKOUT_STEP(27)`) at `packages/fit/src/adapters/messages/messages.mapper.ts`, the SPA editor models only workouts/templates/coaching in Dexie v13, and the MCP server exposes only workout-conversion tools. Users today cannot bring their Garmin health data into Kaiord at all, and AI / planner workflows that should reason about recovery context (HRV trend, last night's sleep, current weight) are forced to treat the athlete as a black box.

This proposal extends KRD to v2.0 so it is the canonical contract for **all** exportable Garmin data, formalises the six format adapters into a partial-coverage matrix (FIT bidirectional for health; TCX / ZWO / GCN workout-only with typed rejection), grows the SPA editor into a Health & Training Hub, and seeds the persistence + MCP surfaces required for downstream consumption. The full ambiguity-gated requirements analysis lives in `.omc/specs/deep-dive-add-health-metrics-to-krd.md` (ambiguity ~12%) and the trace that informed it lives in `.omc/specs/deep-dive-trace-add-health-metrics.md`.

## What Changes

- Extend `fileTypeSchema` (`packages/core/src/domain/schemas/file-type.ts`) with six new variants: `sleep_record`, `weight_measurement`, `hrv_summary`, `daily_wellness`, `body_composition`, `stress_episode`. Total: 9 variants (3 existing + 6 new).
- Make `KRDMetadata.sport` optional (`packages/core/src/domain/schemas/krd/metadata.ts`). Add a Zod refinement that keeps `sport` required for the three legacy types and allows omission only for the six new health types.
- Replace the open `extensions: z.record(z.string(), z.unknown())` with a tagged discriminated union that admits `extensions.health.sleep`, `extensions.health.weight`, `extensions.health.hrv`, `extensions.health.daily`, `extensions.health.bodyComposition`, `extensions.health.stress` while still allowing arbitrary unknown namespaces to round-trip (per the existing extension-preservation rule).
- Add six tagged Zod sub-schemas under `packages/core/src/domain/schemas/health/` (one file per metric, < 100 lines each, kebab-case file names). Each sub-schema describes a single health metric's invariants (e.g., sleep stages SHALL sum to total duration ±1 minute; weight values SHALL be in kilograms; HRV rMSSD SHALL be milliseconds).
- Make `@kaiord/fit` the only bidirectional adapter for health. Register `WEIGHT_SCALE`, `MONITORING`, `MONITORING_INFO`, `SLEEP_LEVEL`, `HRV`, `STRESS_LEVEL`, `BODY_COMPOSITION` in `FIT_MESSAGE_NUMBERS` and the corresponding FIT `file_type` values (9 / 15 / 28 / 32). Route them in `groupWorkoutMessages` to six new bidirectional mappers (`fit-to-krd-health-*.converter.ts` + `krd-health-*-to-fit-*.converter.ts`).
- Formalise `@kaiord/tcx`, `@kaiord/zwo`, `@kaiord/garmin` as workout-only partial adapters. Replace the generic `throw new Error(...)` in their writers with a typed `UnsupportedKrdTypeError` thrown when the input KRD's `type` is one of the six new health variants. Their readers SHALL never emit a health `type`.
- Add `packages/core/docs/ADAPTER-COVERAGE.md` documenting the format × KRD-type matrix and explicitly declaring that cross-format round-trip is required only for the three workout types; health types are FIT-only.
- Extend `DexiePersistenceAdapter` to schema v14 with six new stores that mirror the KRD `extensions.health.*` shape one-to-one: `healthSleep`, `healthWeight`, `healthHrv`, `healthDaily`, `healthBodyComposition`, `healthStress`. Each store keyed on `id` with indexes `[profileId+date]` and `date`. Add six matching repositories (`HealthSleepRepository`, etc.) to `PersistencePort` and to the in-memory adapter. Existing stores carry their v13 schema strings unchanged.
- Extend SPA routing with five new routes under the Health Hub area: `/health` (aggregated dashboard), `/health/sleep`, `/health/weight`, `/health/recovery`, `/health/activity`. Introduce a primary navigation surface (tab bar) bracketing the SPA shell into `Training | Health | Settings` so the new routes are reachable; the existing five workout routes remain inside the Training tab. Each new route is a routed page per the existing SPA surface classification rule, uses `useLiveQuery`, and lands focus on `[data-route-heading]`.
- Extend the FIT import flow in the SPA so it inspects the FIT `file_type` header and routes health files to the health pipeline (KRD `extensions.health.*` → Dexie health store) instead of the workout pipeline.
- Add at least five new MCP tools to `@kaiord/mcp` (`get-health-summary`, `get-sleep-history`, `get-weight-history`, `get-hrv-history`, `get-recovery-status`) and update `SERVER_INSTRUCTIONS` so LLM workflows can read the new data domain.
- Bump KRD to **version 2.0** (`version` field in every produced KRD) and bump `@kaiord/core` plus every adapter that depends on the new shape with a `major` changeset. Document the breaking change and upgrade path in `docs/krd-format.md`.
- Garmin Connect HTTP endpoints for health, bridge-extension scraping of wellness pages, and write-back of weight are **out of scope** for this change and tracked as follow-up issues (listed in `docs/krd-format.md` under "Health follow-ups").

## Capabilities

### New Capabilities

- `health-data`: defines the six health-metric sub-schemas, the `extensions.health.*` namespace, the per-metric invariants, and the contract that the FIT adapter is the only bidirectional implementation. The capability is independent of `krd-format` so that future non-FIT health sources (WHOOP, Oura, Apple Health) can claim conformance without touching the canonical format.

### Modified Capabilities

- `krd-format`: extends the `type` enum from 3 to 9 variants, makes `metadata.sport` conditionally optional, re-tags `extensions` as a discriminated union that includes the reserved `extensions.health.*` namespaces while still preserving unknown namespaces, and bumps the canonical schema version to `2.0`.
- `adapter-contracts`: adds the `UnsupportedKrdTypeError` contract for workout-only writers (TCX / ZWO / GCN), declares `@kaiord/fit` as the only bidirectional adapter for the six health types, and adds the format × type coverage matrix as a normative artefact.
- `spa-persistence-port`: extends `PersistencePort` with six health repositories, adds the Dexie v14 migration plan, and extends the live-query rule to cover the new health entities.
- `spa-routing`: introduces a primary navigation surface (Training / Health / Settings tab bar) and classifies the five new `/health/*` URLs as routed pages, preserving the existing surface-classification invariants (no dual-mount of content components, focus management, live-announcer).

## Impact

- **Code (core)**: `packages/core/src/domain/schemas/file-type.ts` (enum extension), `packages/core/src/domain/schemas/krd/metadata.ts` (conditional `sport`), `packages/core/src/domain/schemas/krd/index.ts` (extensions discriminated union), `packages/core/src/domain/schemas/health/*` (six new sub-schemas + tests), `packages/core/src/domain/schemas/krd/version.ts` (bump to `"2.0"`).
- **Code (FIT)**: `packages/fit/src/adapters/shared/message-numbers.ts` (registration), `packages/fit/src/adapters/messages/messages.mapper.ts` (routing replaces silent discard), six new `*.converter.ts` pairs + tests, six new FIT fixtures.
- **Code (TCX / ZWO / GCN)**: typed `UnsupportedKrdTypeError` thrown from each writer when KRD `type` is health; new error class in `packages/core/src/domain/errors/`.
- **Code (SPA)**: `packages/workout-spa-editor/src/adapters/dexie/dexie-schemas.ts` (CORE_V14), `dexie-migrations.ts` (no-op upgrade), six new `Health*Repository` adapters, `AppRoutes.tsx` (five new routes), new `MainLayout` navigation surface, new `pages/health/*` components, FIT import flow router.
- **Code (MCP)**: `packages/mcp/src/server/create-server.ts` (instructions + tool registration), five+ new tool implementations + tests.
- **Tests**: round-trip tests for the six FIT health types with documented tolerances (sleep stage durations ±1 min, weight ±0.1 kg, HRV rMSSD ±1 ms, daily wellness counts exact); writer-rejection tests for TCX / ZWO / GCN; Dexie migration test v13 → v14; SPA routing tests for the five new routes; MCP smoke tests for the new tools.
- **Docs**: `docs/krd-format.md` (v2.0 changes + health follow-ups), new `packages/core/docs/ADAPTER-COVERAGE.md`.
- **Breaking changes**: yes. KRD `version` advances from `1.0` to `2.0`. External consumers that validate KRD with `metadata.sport` as required will fail on the new health types; documented mitigation is to upgrade to the new Zod schemas from `@kaiord/core@2.0.0`. Changesets are `major` for `@kaiord/core`, `@kaiord/fit`, `@kaiord/tcx`, `@kaiord/zwo`, `@kaiord/garmin`, and `@kaiord/workout-spa-editor` (private but bumped for hygiene).
- **No GCN client / bridge changes**: `@kaiord/garmin-connect`, `@kaiord/garmin-bridge`, and `@kaiord/train2go-bridge` are intentionally untouched — health-API integration is a separate follow-up so this proposal does not balloon.
