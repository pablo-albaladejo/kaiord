## Context

The calendar week view is the SPA editor's home page (`/calendar` and `/calendar/:weekId`) and the canonical surface for both `WorkoutRecord`s (athlete-edited sessions) and `CoachingActivityRecord`s (coach-prescribed activities synced from Train2Go today, TrainingPeaks tomorrow).

Today the two are rendered as **independent stacks** inside `DayColumn` — coaching cards above, workout cards below — with no shared visual language and no domain link. `CoachingActivityCard` paints itself with a rose dashed border and renders the `status` string at the end of an unconstrained `flex gap-2` row, which produces the visible overflow bug where "pending" is painted outside the card. Title is `truncate`-clamped to a single line, so on the typical 140-pixel column the most semantically important field on the card disappears first.

The structural problem behind the visual one is that "what the coach prescribed" and "what the athlete did" describe — most of the time — the same session, but the model has no way to assert that. There is no `matchedWorkoutId` on `CoachingActivityRecord`, no `matchedActivityId` on `WorkoutRecord`, no use case to link them, and consequently no UI to show their relationship. Athletes have to visually pair the two stacks themselves, which is exactly the cognitive load a calendar should remove.

This change addresses both layers in one pass because either alone would leave the screen incoherent — fixing the cards without the match model would still produce two disjoint stacks; introducing the match model without redoing the cards would have nowhere to render the new states.

## Goals / Non-Goals

**Goals:**

- Eliminate the rendering bug in `CoachingActivityCard` (status overflow, title truncation, redundant sport label, rose dashed border).
- Establish a single visual language across `WorkoutCard`, `CoachingActivityCard`, and the new `MatchedSessionCard` — status drives a lateral border colour, the origin (T2G / TP / manual) is a muted text chip, the title is the protagonist.
- Introduce a TrainingPeaks-style **plan ↔ execution match** model: a `CoachingActivity` and a `WorkoutRecord` can be linked into a single rendered session (`MatchedSessionCard`) and unlinked back into two cards.
- Provide `matchSession`, `unmatchSession`, and `autoMatchSessions` use cases. Auto-match is a heuristic-driven **suggestion** (same day + same sport + duration ±20%) confirmed by the user — never a silent mutation.
- Surface a duration-based **compliance hint** on `MatchedSessionCard` (lateral border gradient amber→emerald). Treat richer scoring (zones, TSS) as a clearly-deferred follow-up.
- Add a persisted `calendarDensity` user preference (`compact` ↔ `comfortable`) with a header toggle. Establish `spa-user-preferences` as a capability so future preferences have a home.
- Refine the calendar header (human-readable week label, icon-only sync button with last-sync tooltip) and `DayColumn` (`min-w-[140px]`, today-as-pill, hover-revealed empty-day menu, mobile snap-x).

**Non-Goals:**

- Drag-and-drop between days or between plan/actual cards. The match flow is via a dialog action ("Match to…") and an auto-suggestion banner, not a drag gesture.
- Zone-based or TSS-based compliance scoring. v1 is duration-only; the lateral gradient and any score text reflect duration variance. Richer scoring is out of scope and tracked as a follow-up.
- A new month or list calendar view. The density toggle widens individual cards; it does not change layout topology.
- Redesign of the `EmptyDayDialog` itself. Only its trigger (the `+` affordance and its hover menu) is touched.
- Changes to the FIT/TCX/ZWO/GCN converters or to `@kaiord/core`. The change is scoped to `@kaiord/workout-spa-editor`.
- Changes to bridge protocols, Train2Go extension, or the Garmin Connect adapter.

## Decisions

### D1 — `SessionMatch` is its own aggregate, not a foreign key on either side

**Decision:** persist matches as rows in a new Dexie table `session_matches` with shape `{ id, profileId, coachingActivityId, workoutId, createdAt, source: 'manual' | 'auto' }`, instead of adding `matchedWorkoutId` to `CoachingActivityRecord` and `matchedActivityId` to `WorkoutRecord`.

**Rationale:**

- A `CoachingActivity` is regenerated by every `syncWeek` call and re-upserted by composite key. Storing the match on the activity would force every sync to reconcile the field (dropping it on `delete`, restoring it on re-upsert), which couples sync semantics to match semantics needlessly.
- A `WorkoutRecord` is independent of any coaching platform — many users will never link an account. Adding an optional field to every workout pollutes the type for the majority who don't need it.
- A separate aggregate makes auto-match suggestions (which need to enumerate candidate pairs) a one-table query, and makes "show me all matched sessions this month" trivially expressible.
- Cascade rules become explicit: deleting a coaching activity OR a workout deletes the corresponding `session_matches` row(s) via Dexie cascade hooks. No dangling references on either side.

**Trade-off:** rendering a `MatchedSessionCard` requires a join (read both sides given a match row). Mitigation: the calendar already builds `workoutsByDay` and `coachingByDay` per-day; we add a `matchesByDay` query that returns hydrated `{ activity, workout, score }` objects, and `DayColumn` consumes the three per-day buckets in order. Live-query reactivity (Dexie + `useLiveQuery`) handles invalidation automatically.

**Public-facing on the type:** `CoachingActivity` (the view-model the UI consumes) gains an optional, **derived** `matchedWorkoutId?: string` populated by the join in `useCoachingActivities` — but this is a view-model field, not a stored field on `CoachingActivityRecord`. The persistence boundary stays clean.

**Alternatives considered:**
- *Field on both sides*: rejected for the sync coupling and pollution reasons above.
- *Single field on `CoachingActivity` only*: rejected because a workout without a matching activity (athlete uploads independently of any coach) can never offer "match this to a plan" without enumerating activities anyway.

### D2 — Auto-match is a confirmable suggestion, not a silent mutation

**Decision:** `autoMatchSessions(profileId, weekStart)` returns a list of `MatchSuggestion` objects (each `{ activityId, workoutId, score, reason }`) without writing anything. The calendar surfaces suggestions as a dismissable banner ("Pair these 3 sessions?") with per-row Accept/Reject. Only on Accept does the use case write a `session_matches` row.

**Rationale:** matching is irreversible-feeling for users even though `unmatchSession` exists — once two cards become one, the cognitive cost of un-fusing them is higher than the cost of confirming the fusion. Silent automatic matching also makes incorrect matches invisible (the user never sees the planned and actual cards, so they can't notice the heuristic was wrong).

**Heuristic for v1:** `score = 1 - |plan.duration - actual.duration| / plan.duration`, gated by `same date AND same sport`. Suggestions surface when `score >= 0.6` (i.e., ≤40% variance). Multiple workouts on the same day in the same sport produce one suggestion per highest-score pair, with no overlapping pairings (greedy assignment).

**Alternatives considered:**
- *Silent auto-match on sync*: rejected — incorrect matches become invisible, and re-sync semantics (does the auto-match re-run? does it overwrite manual matches?) become a pile of edge cases.
- *Auto-match with toast undo*: rejected — toast disappears, the user moves on, the wrong match persists. Confirmation up front is cleaner than reversal after the fact.

### D3 — Status drives the lateral border colour; origin is a muted text chip

**Decision:** every card variant (`CoachingActivityCard`, `WorkoutCard`, `MatchedSessionCard`) uses `border-l-4` (4 pixels — wide enough to qualify as graphical, not incidental, under WCAG 1.4.11) in a status-derived colour: `amber-600` for pending, `emerald-600` for completed, `slate-500` for skipped. For `MatchedSessionCard`, a CSS gradient (amber-600 → emerald-600) on the left edge encodes the compliance score. Origin (`T2G`, `TP`, `manual`) renders as `text-[10px] text-slate-500` at the bottom of the card with a leading `· ` separator — never a coloured badge.

**Rationale:**

- Status is the most-asked question per session; making it the strongest visual signal answers it at a glance.
- Origin is a metadata answer to "where did this come from", which the user only cares about when something is wrong. A muted chip respects that priority.
- The lateral border is bandwidth-cheap, accessible at distance, and works under both light and dark themes without re-tinting the card body. It also avoids the "rose dashed = warning" misread of today's design.
- The selected colour tokens (`amber-600 #d97706`, `emerald-600 #059669`, `slate-500 #64748b`) achieve **≥ 3:1** contrast against a white card body (Tailwind v3 reference values: amber-600 4.5:1, emerald-600 3.4:1, slate-500 4.6:1), satisfying WCAG 1.4.11 (Non-text Contrast) for graphical UI components.

**Trade-off:** the lateral border alone is **not** the WCAG-conformant signal channel — the **status icon** (Clock / Check / Minus, with `aria-label`) is the canonical channel that satisfies WCAG 1.4.1 (Use of Color). The border is a redundant visual cue. The icon renders in every density mode; in `comfortable` mode the status text label reinforces it. Colour-blind users (deuteranopia, protanopia) get full information from the icon alone. Verified via Storybook a11y addon (zero violations) and a contrast-measurement task in the test plan.

### D8 — Compliance score has explicit bucket boundaries and a numeric reading

**Decision:** the lateral gradient on `MatchedSessionCard` maps `complianceScore` to three discrete buckets with closed/open interval semantics: `[0, 0.5)` → amber-600, `[0.5, 0.8)` → mid (CSS gradient amber-600 → emerald-600 sampled at the midpoint, `yellow-700` per the contrast-test task 4.5), `[0.8, 1.0]` → emerald-600. `null` (missing duration on either side, OR `MatchSuggestion.score === null` which has the same semantic) → `slate-400` (neutral). The card SHALL also render a tooltip (and an `aria-label` on the card root) of the form `"Matched session — <percent>% (<actualDur>/<plannedDur>)"`. In `comfortable` density the percentage SHALL render as on-card text (`text-[10px] text-slate-600`) so sighted users get a numeric reading without hovering.

**Rationale:**

- A pure gradient cannot communicate "0.7 vs 0.8" — the precision the user needs to act on the data is lost.
- Discrete buckets keep the visual encoding deterministic (no implementation drift) and aligned with the score derivation tests.
- The tooltip + `aria-label` make the score available to screen readers and to sighted-user hover, satisfying WCAG 1.4.1 for the compliance signal (which is otherwise colour-only).

### D9 — `useMatchedSessions` is a single live query plus two `bulkGet` reads

**Decision:** the `useMatchedSessions(profileId, days)` hook performs exactly one `useLiveQuery` over `session_matches` filtered by `profileId` and the day range, then issues two `bulkGet` calls — one to `coachingActivities.where('id').anyOf([...activityIds])` and one to `workouts.where('id').anyOf([...workoutIds])` — to hydrate the matched sides. The compliance score is computed in the hook's projection step.

**Rationale:**

- A naïve per-row join would be `n` separate reads per render. With `bulkGet`, we get O(2) reads per re-render regardless of match count.
- `useLiveQuery` reactivity on `session_matches` covers create/delete on the match side; the underlying activity/workout updates are picked up because the same calendar page already runs `useLiveQuery` on those tables for the solo card buckets — when those queries re-fire, the page rebuilds and the `bulkGet` results are re-resolved.
- This bounds the worst-case calendar render at 3 live queries plus 2 bulk reads per re-render.

**Trade-off:** an activity/workout updated in place (e.g., title edit) does not directly trigger `useMatchedSessions` to re-run — it relies on the parent live queries firing. Acceptable: the calendar page already subscribes to both tables for the solo-card buckets, so the parent re-render is guaranteed. A perf budget (D11) backs this up.

### D10 — SPA package adopts a port/adapter layering for `SessionMatch` and `UserPreferences`

**Decision:** `packages/workout-spa-editor/src/` gains explicit `ports/` and `adapters/` (Dexie) directories for the new aggregates, mirroring the convention used in `@kaiord/core`. This is the first time the SPA package adopts visible hexagonal layering at the source-tree level (existing SPA code intermixes layers).

**Rationale:**

- The new aggregates introduce port-shaped seams (cascade hooks, in-memory test doubles, possible future remote backing) that benefit from explicit layering.
- Future SPA-internal aggregates can follow the same pattern incrementally rather than via a big-bang refactor.
- Documented here so future readers don't re-litigate the structure.

**Trade-off:** a partial layering (some aggregates in `ports/`, some not) can confuse newcomers. Acceptable for v1; future changes can promote existing aggregates as they're touched.

### D11 — Performance budget for the redesigned calendar

**Decision:** the redesigned `CalendarPage` SHALL render its first contentful paint within **200 ms** on a mid-tier device (Moto G Power 2022 reference, 4× CPU throttling in DevTools), measured against a synthetic week containing 30 cards distributed as 10 matched, 10 solo plans, and 10 solo actuals. The `useMatchedSessions` hook SHALL not contribute more than **30 ms** to that budget. These targets are recorded as acceptance criteria for the implementation tasks.

**Rationale:**

- The current calendar is fast; introducing a join risks regression. A budget makes regressions visible at task-completion time, not at user-report time.
- The 30-card synthetic week is representative of an active athlete using both T2G and manual entry.

### D12 — Forward-only deploy with service-worker cache invalidation

**Decision:** the Dexie schema bump is **forward-only**. Once a user's browser opens the new bundle and Dexie applies the schema migration, the same browser SHALL NOT successfully open an older bundle against the migrated schema. The deploy SHALL invalidate the service-worker cache so users reliably load the new bundle. The release notes SHALL state this constraint: "rolling back this release after deploy requires either a forward-fix release or accepting that already-migrated users will see a `VersionError` on the older bundle".

**Rationale:**

- Dexie schema versions are monotonic — there is no built-in downgrade path.
- A forward-only commit is honest about the trade. The alternative (versioned data migrations with downgrade adapters) is significantly more work for an SPA-internal change with no backend coupling.
- Service-worker cache invalidation prevents the most common downgrade scenario (stale cached bundle).

**Trade-off:** rollback complexity in case of a P0 bug. Mitigation: gated behind a viewport-noticeable visual change (the new card layout) so QA in staging catches issues before promotion; staged rollout via the existing release pipeline; immediate forward-fix lane if a regression slips through.

### D13 — Motion respects `prefers-reduced-motion`

**Decision:** every animated surface introduced by this change (sync-button spinner, banner slide-in, hover-revealed empty-day menu, mobile snap scroll) SHALL respect `@media (prefers-reduced-motion: reduce)`. Animations SHALL collapse to instant state changes; the mobile scroll SHALL drop `snap-mandatory` in favour of `snap-proximity` (which assistive-tech tools handle gracefully) under reduced motion AND unconditionally for compatibility with VoiceOver / TalkBack.

**Rationale:**

- WCAG 2.3.3 (Animation from Interactions) requires reduced-motion support for non-essential animation.
- `snap-proximity` avoids the assistive-tech focus-trap behaviour observed with `snap-mandatory` while still settling the scroll cleanly on a day in 90% of swipes.

### D4 — Density preference is its own capability (`spa-user-preferences`)

**Decision:** introduce a `spa-user-preferences` capability with a `UserPreferencesRepository` port and a single Dexie table `user_preferences` keyed by `profileId`. v1 stores `{ profileId, calendarDensity: 'compact' | 'comfortable' }`. Use cases: `getUserPreferences(profileId)`, `setCalendarDensity(profileId, density)`.

**Rationale:**

- The current SPA has no home for user preferences. Folding density into `spa-calendar` would force every future preference (theme override, default sport filter, week-start day, …) to either accrete in the calendar capability or be retroactively extracted.
- Per-profile scoping matches the pattern established by `spa-coaching-integration` (linked accounts are per-profile), so multi-profile households (parent / child sharing the device) can have different defaults.
- Defaults are computed at read time from viewport width, not stored, so a freshly-created profile doesn't need a preferences row to render correctly. The row is created on first user-driven mutation.

**Alternatives considered:**
- *localStorage*: rejected — not per-profile, doesn't survive profile switches cleanly, doesn't share the Dexie reactivity story used elsewhere in the SPA.
- *Zustand + manual persist*: rejected per the project rule "Zustand only for editor runtime; persisted data goes to Dexie".

### D5 — Three card components, not one polymorphic card

**Decision:** ship three distinct components — `CoachingActivityCard` (SOLO PLAN), `WorkoutCard` (SOLO ACTUAL), `MatchedSessionCard` (MATCHED) — sharing a private layout primitive (`CardShell`) that owns the lateral border, the title row, and the metadata row.

**Rationale:**

- The three states display different data (plan-only shows prescribed metrics; actual-only shows executed metrics; matched shows both with compliance), so a single polymorphic component would carry three sets of optional fields and three branches in its JSX. The 60-line per-component limit (per `CLAUDE.md`) makes the polymorphic version harder to keep readable than three focused ones.
- Shared visual language is enforced by the shared `CardShell`, not by component-level branching.
- Test surface is per-state, mirroring how Storybook stories are organised.

**Trade-off:** three test files instead of one. Acceptable; each is small and focused.

### D6 — Compliance score derivation is a pure function; the view-model hook composes it

**Decision:** the score is computed by a pure application-layer function `computeComplianceScore(planDur: number | undefined, actualDur: number | undefined): number | null` in `packages/workout-spa-editor/src/application/compute-compliance-score.ts`, returning `clamp(1 - |planDur - actualDur| / planDur, 0, 1)` when both durations are present and `null` otherwise. The `useMatchedSessions` view-model hook composes this function with the duration parser (D6a) and the per-row join. Bucket boundaries (D8) drive the visual encoding in `MatchedSessionCard` — they are NOT baked into the score.

The pure function has its own unit tests (boundary inputs, missing inputs, NaN guards). The hook has separate tests for the join + projection.

**Rationale:**

- Splitting the pure derivation from the view-model hook satisfies hexagonal layering: domain/application stays viewport-agnostic; the React hook is infra.
- Future scoring (zones, TSS) replaces `computeComplianceScore` without touching the hook signature.

**Trade-off:** a workout that hit the wrong duration but the right zones (perfect aerobic adherence at the wrong time) reads as poor compliance. Acceptable for v1; richer scoring is the explicit follow-up.

### D6a — Coaching duration is a free-text string; parsing is an explicit application function

**Decision:** `CoachingActivityRecord.duration` is typed `string | undefined` (free-text from the platform — e.g., `"45 min"`, `"1h 30"`, `"45'"`, `"~45min"`, `"45-50 min"`, `"PT45M"`). A pure application function `parseCoachingDuration(s: string | undefined): number | undefined` in `packages/workout-spa-editor/src/application/parse-coaching-duration.ts` SHALL parse it to seconds, returning `undefined` on parse failure. The function MUST handle the canonical Train2Go forms `"<n> min"`, `"<n>h"`, `"<n>h <m>"`, `"<n>h<m>"`, `"<n>'"`, `"PT<n>M"`, and ISO 8601 durations (`"PT1H30M"`); MUST tolerate leading `~` (approximate marker, dropped); MUST treat ranges (`"45-50 min"`) as their lower bound; MUST return `undefined` on any other input.

`computeComplianceScore` consumes the parsed seconds. When either side is `undefined`, the score is `null` and the card renders the neutral grey lateral border (per D8).

**Rationale:**

- Without an explicit parser, every consumer would invent its own and they would drift.
- Free-text duration is a known, irreducible quirk of the coaching APIs — making the parser an explicit, tested artifact (rather than a hidden inline regex) is the only sustainable answer.
- The lower-bound rule for ranges is a deliberate UX choice: users are more likely to be irritated by a "missed by 50%" reading on a low-effort day than by a "hit 100%" reading on the upper bound.

**Trade-off:** users who consistently train at the upper bound of a planned range get a slightly unfavourable compliance score. Acceptable; documented in the spec as the v1 convention.

### D7 — Header sync button: icon-only with tooltip

**Decision:** `CoachingSyncButton` becomes an icon button (`RefreshCw` from lucide-react) sized 32×32 with a tooltip showing `{label} · last sync {relative-time}` (e.g., "Train2Go · 12m ago"). The spinner replaces the icon in-place during sync. The not-connected state retains a text "Connect to {label}" button (low-frequency, deserves discoverability).

**Rationale:**

- For a connected, normally-functioning sync, the action is rare (auto-sync handles 95% of cases); the button doesn't deserve to grab attention away from the calendar grid.
- Last-sync timestamp answers the "did this just sync?" question the user actually asks when looking at the button.
- The not-connected state is exceptional and deserves the bigger affordance.

## Risks / Trade-offs

- [**Risk** — auto-match heuristic produces wrong suggestions on busy days] → Mitigation: confirmation flow (D2) keeps the user in control; greedy assignment with `score >= 0.6` threshold and "no overlapping pairings" rule reduces false positives; the `SessionMatchSuggestion` is dismissable per-row, not all-or-nothing.
- [**Risk** — Dexie schema migration corrupts existing calendars] → Mitigation: the migration adds two empty tables and one non-breaking index; no existing rows are mutated; the change is forward-compatible (older code that doesn't know about `session_matches` simply doesn't read it). Migration tested with a round-trip fixture loading pre-change Dexie state and asserting all existing reads succeed.
- [**Risk** — three card components diverge over time as small changes land in one but not others] → Mitigation: shared `CardShell` primitive owns the lateral border and metadata row layout; each card composes it. A regression test asserts all three render the lateral border with the same width/colour token for the same `status`.
- [**Risk** — `MatchedSessionCard` cognitive load (two rows in one card) on small viewports] → Mitigation: the `compact` density mode collapses `planned/actual` into a single row showing only the `actual` duration with a lateral compliance gradient (the planned data is in the dialog). The `comfortable` mode shows both rows.
- [**Risk** — density preference races with profile switch (read for old profile applies briefly to new profile)] → Mitigation: `useUserPreferences(profileId)` is a `useLiveQuery` keyed on `profileId`; switching profiles triggers a re-query, and density falls back to the viewport-based default if no row exists yet.
- [**Risk** — colour-only status indication fails for colour-blind users] → Mitigation: status icon (Clock / Check / Minus) is always rendered in `compact` mode; status text is rendered in `comfortable` mode. Colour is a redundant channel.
- [**Trade-off** — match aggregate adds query complexity for the 95% case (single-profile, manually managed calendar)] → Accepted: the join lives in one hook (`useMatchedSessions`), Dexie's query planner handles it efficiently, and the mental clarity of the model justifies the cost.
- [**Trade-off** — duration-only compliance score will sometimes mislead] → Accepted explicitly; documented in the proposal and in the spec scenario for compliance scoring as the v1 floor, with richer scoring tracked as a follow-up.
