# Strava

Deep research on the **cookie / web-session** approach to Strava integration (reusing the user's `_strava4_session`), plus the official `/api/v3` OAuth API for reference. Scope: the two goals — (1) session persistence/renewal, (2) endpoint catalog (read + write). The registered-developer OAuth app path is explicitly **off the table** except as a reference column.

Bottom line up front:

- The durable web credential is **`strava_remember_token`** (a JWT set when "remember me" is on), not `_strava4_session`. The session cookie is short-lived/rotating; the remember-token silently re-mints it. In a browser/extension, the cookie jar handles this for you; server-side you must capture and replay the JWT.
- Rich, **undocumented JSON endpoints** exist under `www.strava.com` (notably `GET /athlete/training_activities`, `/activities/{id}/streams`, `/activities/{id}/export_gpx`) and are used by real extensions (SauceLLC, Elevate) that replay the web session. Writes exist too (kudos, flag, delete).
- **ToS risk is real and rising.** Strava's Terms prohibit automated/scraped access, and as of **June 2026** Strava is actively auth-walling data, retiring API endpoints, charging developers, and (per its CEO) hunting scrapers ahead of its IPO. Reusing the web session for automation is a gray-to-red area; writes and high request rates are the most enforced.

## Session persistence & renewal

**The cookies.** Logging in (`POST /session` with `remember_me=on`) sets three relevant cookies:

- **`_strava4_session`** — the Rails encrypted session cookie. `HttpOnly`, `Secure`, domain `.strava.com`. Short-lived / rotating; by default a _session_ cookie (dies on browser close) — it is **not** a stable long-term credential.
- **`strava_remember_id`** — the athlete/web id (used to build the `/me` redirect target).
- **`strava_remember_token`** — a **JWT** (`header.payload.signature`); this is the durable "remember me" persistent-login credential (Rails/Devise-style).

**Renewal model (the key insight).** `_strava4_session` is disposable; the JWT `strava_remember_token` is what re-mints it. stravaweblib proves you can rebuild an authenticated session from **only** the JWT ([`webclient.py` `_login_with_jwt`](https://github.com/pR0Ps/stravaweblib/blob/master/stravaweblib/webclient.py)):

1. Decode the JWT middle segment (base64) to recover the web/athlete id.
2. Set cookies `strava_remember_id` and `strava_remember_token` on `.strava.com` (Secure).
3. `GET https://www.strava.com/me` — Strava responds with a redirect to `/athletes/{web_id}`, and in doing so re-issues a fresh `_strava4_session`.

So: **store the JWT, not the session cookie.** `stravaweblib` exposes it via a `jwt` property (`self._session.cookies.get('strava_remember_token')`), and the README states plainly that "storing this JWT … allows for resuming the session without having to log in again."

**In a browser extension (our case).** A service worker doing `fetch('https://www.strava.com/...', {credentials:'include'})` doesn't need to manage any of this: the browser cookie jar holds `_strava4_session`, and if the user ticked "remember me", the `strava_remember_token` automatically re-mints the session cookie as it rotates. You only need to capture/replay the JWT yourself when running **server-side / headless** (no persistent cookie jar).

**Login/CSRF flow (if you must log in programmatically).** It's Rails:

1. `GET /login` (or `/about`) → parse hidden `<input name="authenticity_token">` and/or `<meta name="csrf-token">`.
2. `POST /session` with `utf8`, `authenticity_token`, `email`, `password`, `remember_me=on`.
3. Success = 302 redirect **not** back to `/login`; cookies are now set.
4. For subsequent **writes**, send the CSRF token as header **`x-csrf-token`** (value from `<meta name="csrf-token">`). JSON reads also expect `X-Requested-With: XMLHttpRequest` and a browser-like `User-Agent`.

**What breaks the session:**

- User logs out (invalidates `strava_remember_token`), or changes password.
- Strava rotates the session secret / JWT signing keys, or the JWT simply expires (lifetime is **not publicly documented**).
- Cookie eviction / browser close without remember-me (session cookie only).
- A Cloudflare / anti-bot challenge is interposed (none observed today by the komoot-to-gpx spike, but Strava is actively hardening — see below).
- Explicit deauthorization or account action by Strava.
- Note: this is **separate** from the OAuth path, where the _access token_ expires every **6 hours** and the _refresh token_ lasts until used/deauthorized — those numbers do **not** apply to the web session.

## ToS risk

Concrete and currently escalating. Treat web-session automation as gray-to-red.

- **Terms of Service** ([strava.com/legal/terms](https://www.strava.com/legal/terms)) prohibit "web scraping, web harvesting, web data extraction … or any other automated means to extract data," and prohibit any "robot, spider, offline reader" that sends "more request messages … than a human can reasonably produce in the same period of time using a conventional web browser." Reusing the web session with an automated client falls squarely under these clauses even when it's the user's own data.
- **API Agreement** ([press.strava.com — Updates to Strava's API Agreement](https://press.strava.com/articles/updates-to-stravas-api-agreement)) bans using Strava data (even publicly viewable, even aggregated/de-identified) for analytics or AI-model training — the AI-training ban took effect **November 2024**.
- **June 2026 crackdown** ([TechCrunch: "Strava declares war on scrapers ahead of IPO"](https://techcrunch.com/2026/06/01/strava-declares-war-on-scrapers-ahead-of-ipo/), corroborated by [Neowin](https://www.neowin.net/news/strava-tightens-api-access-in-bid-to-fend-off-data-scraping-ai-companies/), [TechRepublic](https://www.techrepublic.com/article/news-strava-api-scraping-crackdown/)): ahead of a confidential IPO filing, Strava is (a) **auth-walling** data that used to be viewable without login, so "only authenticated users can view certain data"; (b) **retiring some API endpoints** (90-day grace); (c) introducing a **$11.99/month** developer fee; (d) adding MCP support for tighter control. The CEO singled out Perplexity for trying to circumvent restrictions via aggregators.
- **Practical exposure for the cookie approach:**
  - _Reads of your own activities_ (list/streams/GPX export) at human-like rates are the lowest-risk slice, but still technically violate the automation clauses.
  - _Writes_ (kudos, flag, delete) are the **most enforced** — kudos/flag spam is exactly what Strava polices; automating them risks account suspension.
  - Fragility risk is as important as legal risk: because it rides on undocumented endpoints and a rotating session cookie, Strava can break it at any time (endpoint changes, CSRF/header requirements, Cloudflare challenge, cookie rotation) with no notice or SLA.
  - Mitigations others use: strict **client-side rate limiting** (SauceLLC caps streams at ~30/min, 200/hr, 700/day), browser-like `User-Agent` + `X-Requested-With`, honoring `has_latlng` before exporting, and never batching writes.

### B) Official `/api/v3` endpoints (reference only — OAuth2 Bearer token, base `https://www.strava.com/api/v3`)

**Note for a Tanita→Garmin/Strava sync use-case:** the official write path you'd want is **`POST /api/v3/uploads`** (GPX/TCX/FIT) then poll `GET /uploads/{id}` — but that needs a registered app. There is **no clean internal web-session _upload_ endpoint** in the prior art (the web uploader is a heavier multipart/JS flow); internal writes seen in the wild are limited to delete, kudos, and flag. So for writing activities, the OAuth `/uploads` API is the realistic route; the web session is best for **reads/exports**.
